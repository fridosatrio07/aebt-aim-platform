# UI UX Rules

## Core Principles

- Role-based landing: each user lands in a workbench aligned to role and access scope.
- My Work / Action Inbox: daily work starts from tasks and actions, not passive module browsing.
- Task-first, module-second: module navigation exists, but operational flow points users to work items.
- Batch-first operation: imports, validation, certificate updates, document tagging, evidence selection, assignment, and administrative review should support batch actions.
- Progressive data enrichment: minimum data can start onboarding; incomplete data must show Data Gap, Pending Validation, or Limited Basis states.
- Risk-prioritized workflow: high-risk, overdue, missing-evidence, pending-review, and critical items must be easy to find.
- Approval by exception: layered approvals apply to technical/compliance/risk/certification/evidence export/configuration baseline items; low-risk administrative changes use validation/audit controls.
- Evidence reuse: one document/evidence object can be linked across modules without repeated upload.
- Dashboard-to-action: KPI cards and badges must open filtered action lists.
- Progressive disclosure: first layer shows operational essentials; details, evidence, audit trail, and history open in tabs/drawers/detail pages.

## Page and Component Patterns

- Use table + drawer pattern for high-volume list work.
- Large tables need default saved views and visible active filters.
- Provide empty, loading, error, access denied, and permission-limited states.
- Provide search/filter/sort/pagination for high-volume lists.
- Show tenant/project/site/facility context clearly.
- Do not label draft/preliminary outputs as final, approved, certified, safe, layak operasi, or fit-for-operation unless reviewed/approved by authorized personnel and source-approved.
